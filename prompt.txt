## Requirements

### Libraries
- THE SYSTEM SHALL use `requests` for all HTTP calls.
- THE SYSTEM SHALL use `websockets` for the WebSocket connection.
- THE SYSTEM SHALL NOT use `aiohttp` for any purpose.

### 1. CLI
- THE SYSTEM SHALL accept `--challenge-id` OR `--challenge-name` (mutually exclusive, at least one required), `--name` (required), `--owner` (optional, default `""`), and `--server` (optional, default `https://battleships.devrel.hny.wtf`).
- THE SYSTEM SHALL error if neither `--challenge-id` nor `--challenge-name` is provided.

### 2. Registration & Credential Persistence
- THE SYSTEM SHALL load credentials from `.battleships-registration.json` keyed by `challenge_id` or `challenge_name` (whichever was provided). If found, skip registration.
- WHEN no saved credentials exist THE SYSTEM SHALL POST to `/api/v1/challenges/{challengeId}/bots` (when `--challenge-id` is given) or `/api/v1/challenges/by-name/{challengeName}/bots` (when `--challenge-name` is given), URL-encoding the path segment, with body `{"displayName": name, "owner": owner}`.
- THE SYSTEM SHALL save the full registration response into `.battleships-registration.json` under the challenge key after a successful registration.
- THE SYSTEM SHALL read `authToken` from saved credentials with `.get("authToken")` — it may be absent on reconnect.

### 3. WebSocket Connection
- THE SYSTEM SHALL force `wss://` by replacing everything before `://` in the server-provided `webSocketUrl`.
- WHEN `authToken` is non-empty THE SYSTEM SHALL set the `Authorization: Bearer {authToken}` header AND append `?token={authToken}` to the URL.
- WHEN `authToken` is empty or absent THE SYSTEM SHALL NOT set the header or append the query param.
- THE SYSTEM SHALL connect using `websockets.connect(url, additional_headers=headers)`.
- THE SYSTEM SHALL loop over incoming messages until `handle()` returns `False`, then stop. There is no reconnect logic.

### 4. Message Dispatch
- THE SYSTEM SHALL extract `messageType` as `msg.get("messageType", msg.get("type", "UNKNOWN"))`.
- THE SYSTEM SHALL extract `payload` as `msg.get("payload", msg)` to handle both nested and flat server formats.
- THE SYSTEM SHALL handle: `CHALLENGE_REGISTERED`, `PLACE_SHIPS_REQUEST`, `FIRE_REQUEST`, `GAME_RESULT`, `CHALLENGE_COMPLETE`, `ERROR`, and an unhandled fallback.
- THE SYSTEM SHALL return `False` from the handler only on `CHALLENGE_COMPLETE`, and `True` for all other message types.

### 5. Ship Placement
- THE SYSTEM SHALL use a hardcoded ship list: CARRIER (size 5), BATTLESHIP (size 4), CRUISER (size 3), SUBMARINE (size 3), DESTROYER (size 2).
- THE SYSTEM SHALL ignore the ships list from the server request payload.
- THE SYSTEM SHALL place all ships horizontally (`orientation: "H"`) starting at `col: 0`.
- THE SYSTEM SHALL maintain an `occupied` set and check all 8 neighbors (dr, dc in {-1,0,1}×{-1,0,1}) for conflicts before placing each ship.
- THE SYSTEM SHALL always advance `row += 2` after a successful placement, regardless of `allowAdjacentShips`.
- THE SYSTEM SHALL identify each placement with `typeId` (not `shipId` or `id`).
- WHEN a `PLACE_SHIPS_REQUEST` is received THE SYSTEM SHALL initialize a new per-game state dict and send a `PLACE_SHIPS_RESPONSE`:
```json
{"messageType": "PLACE_SHIPS_RESPONSE", "payload": {"gameId": "...", "response": {"placements": [...]}}}
```

### 6. Firing Strategy
- THE SYSTEM SHALL initialize a per-game `target_queue` as all even-parity cells `(r+c) % 2 == 0` (shuffled) followed by all odd-parity cells (shuffled).
- THE SYSTEM SHALL maintain a per-game `shots_fired` set and `hunt_queue` list.
- WHEN syncing shots from `opponentView.shots`: for each shot not already in `shots_fired`, add it and — if result is `"HIT"` — insert all valid in-bounds unfired non-queued orthogonal neighbors at the front of `hunt_queue`.
- WHEN choosing a target: drain `hunt_queue` (skip already-fired), then fall through to `target_queue` (skip already-fired), then exhaustive row/col scan as fallback.
- WHEN a `FIRE_REQUEST` is received THE SYSTEM SHALL sync shots, choose a target, and send a `FIRE_RESPONSE`:
```json
{"messageType": "FIRE_RESPONSE", "payload": {"gameId": "...", "response": {"target": {"row": r, "col": c}}}}
```
- WHEN a `FIRE_REQUEST` arrives for a game with no existing state THE SYSTEM SHALL initialize a new game state for it.

### 7. Game Lifecycle
- WHEN `PLACE_SHIPS_REQUEST` is received THE SYSTEM SHALL store a new game state under `gameId`.
- WHEN `GAME_RESULT` is received THE SYSTEM SHALL remove the game state for `gameId`.
- WHEN `CHALLENGE_COMPLETE` is received THE SYSTEM SHALL print the summary and return `False` to stop the loop.
- THE SYSTEM SHALL NOT attempt to reconnect after the WebSocket closes.
- THE SYSTEM SHALL catch `websockets.exceptions.ConnectionClosedError` and exit cleanly with a message, as the server may drop the connection without sending a close frame.
- THE SYSTEM SHALL accept a `--timeout` argument (integer seconds, default 60) and cancel the WebSocket session after that duration, printing a timeout message and exiting cleanly.

### 8. Payload Field Names
- `PLACE_SHIPS_REQUEST`: `payload.request.gameId`, `payload.request.rules.allowAdjacentShips`, `payload.request.opponentView` — with fallback to `payload.gameId` for flattened payloads.
- `FIRE_REQUEST`: `payload.request.gameId`, `payload.request.turnNumber`, `payload.request.opponentView`.
- `GAME_RESULT`: `payload.gameId`, `payload.opponentDisplayName`, `payload.isWin`, `payload.playerHits`, `payload.opponentHits`, `payload.points`, `payload.gameNumber`, `payload.totalGames`.
- `CHALLENGE_COMPLETE`: `payload.wins`, `payload.losses`, `payload.totalPoints`, `payload.hitDifference`, `payload.isNewBest`.
- `ERROR`: `payload.errorCode`, `payload.message`.

## Design

### Architecture
Single-file async bot. Uses synchronous `requests` for registration, then a single `websockets` async loop for the challenge. Per-game state is stored in a dict keyed by `gameId`. No reconnect logic — the process exits when `CHALLENGE_COMPLETE` is received or the WebSocket closes.

### API

**Base URL:** `https://battleships.devrel.hny.wtf/api/v1`

**Registration endpoints:**
```
POST /api/v1/challenges/{challengeId}/bots
POST /api/v1/challenges/by-name/{challengeName}/bots
Body:     {"displayName": "...", "owner": "..."}
Response: {"registrationId": "...", "challengeBotId": "...", "authToken": "...", "webSocketUrl": "ws://..."}
```

**WebSocket message envelope:**
```json
{"messageType": "...", "payload": {...}}
```

**Message flow:**
```
Connect → CHALLENGE_REGISTERED → per game: PLACE_SHIPS_REQUEST → PLACE_SHIPS_RESPONSE → (FIRE_REQUEST → FIRE_RESPONSE)* → GAME_RESULT → CHALLENGE_COMPLETE
```

**PLACE_SHIPS_REQUEST payload:**
```json
{"request": {"gameId": "...", "board": {"width":10,"height":10}, "ships": [...], "rules": {"allowAdjacentShips": false}}}
```

**PLACE_SHIPS_RESPONSE payload:**
```json
{"gameId": "...", "response": {"placements": [{"typeId": "CARRIER", "start": {"row": 0, "col": 0}, "orientation": "H"}]}}
```
- Use `typeId` as the ship identifier. Do NOT use `shipId` or `id`.
- Always use a 2-row gap between ships. Do NOT use a 1-row gap even when `allowAdjacentShips` is true — the server enforces strict non-adjacency regardless.

**FIRE_REQUEST payload:**
```json
{"request": {"gameId": "...", "turnNumber": 3, "opponentView": {"shots": [{"row": 0, "col": 0, "result": "MISS"}]}}}
```

**FIRE_RESPONSE payload:**
```json
{"gameId": "...", "response": {"target": {"row": 5, "col": 7}}}
```

### Constraints
- Coordinates are 0-based (0–9 on a 10×10 board).
- Ship placement uses `col: 0` and horizontal orientation. The occupied-set conflict check covers all 8 neighbors.
- Credential file `.battleships-registration.json` is a dict of dicts keyed by challenge ID or name, not a flat credential object.
- `global SERVER_URL` must be declared at the top of `main()` before any reference to it as a default value.
